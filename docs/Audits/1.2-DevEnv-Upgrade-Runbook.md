# Satisfactory 1.2 — Dev-Environment Upgrade + Port Runbook

Concrete, ordered runbook for moving SmartFoundations from the 1.1 dev env (UE 5.3 / SML 3.11)
to 1.2 (UE 5.6.1 / SML 3.12). Companion to the strategy in
[`Upgrade-1.2-RefactorPlan.md`](Upgrade-1.2-RefactorPlan.md). Started 2026-05-31, ahead of the 1.2
stable release (2026-06-02). Building against the **Experimental** branch + headers, as Robb (SML)
advised; SirDigby is already doing the same.

**Authoritative sources (re-read often — both change frequently):**
- Migration guide: docs-dev.ficsit.app `.../Development/UpdatingFromSml311.html` ("Upgrading from SML 3.11.x to 3.12.x")
- Full diff: `github.com/satisfactorymodding/SatisfactoryModLoader/compare/v3.11.3...dev`
- SML 1.2 repo: `origin/dev` (EngineAssociation **5.6.1-CSS**; not yet version-tagged 3.12)
- Watch: Discord **#modding-info-feed** (doc-change notifications) and **#toolkit-updates** (the "you can release" signal — not posted as of 2026-05-31).

> **Reality check:** SML 3.12 / 1.2 support is *not released*. SMM can't distribute GameFeature mods
> yet (support is being written; that's the gate on 3.12's public release). We are building locally
> against EXP, expecting churn and redownloads until #toolkit-updates posts. Do not publish a 1.2
> build until then.

> **Will Smart! survive uncompiled? No.** The guide's "won't survive" list is: includes C++ modules,
> interfaces with 1.2-changed systems, works with user input — Smart! is all three. Full port required.

## Part A — Install the 1.2 dev environment

Follow SML's "Updating your Mod" guide for exact download links; the dependency set is:

1. **Unreal Engine 5.6.1-CSS** — CSS custom engine release (replaces 5.3.2-CSS).
2. **New clang toolchain** — 5.6.1 requires an updated clang; install per the Dependencies page.
3. **Wwise update** — newer Wwise version; integrate into the project. Versions seen in #sml-development:
   SDK `2023.1.14.8770`, UE integration `2023.1.14.3555`. If the Wwise Launcher misbehaves, `wwise-cli`
   works: `wwise-cli download --sdk-version "2023.1.14.8770" --filter Packages=SDK ...` then
   `wwise-cli integrate-ue --integration-version "2023.1.14.3555" --project "./FactoryGame.uproject"`.
   On first editor open you may see "GeneratedSoundBanks folder does not seem to be set" — open the
   Wwise editor and **generate sound banks once** (one-time unless you add Wwise systems).
4. **.NET 8.0** (was .NET 6 era) — Alpakit/automation targets net8.0 now.
5. **Starter project** — pull the updated FactoryGame starter project from `origin/dev` (engine assoc
   `5.6.1-CSS`). Keep a backup of the current 1.1 project (the guide needs the old project to recover
   Enhanced Input field values — see B/port).
6. **Satisfactory game — Experimental branch** (game CL **480321** as of 2026-05-31). Mods don't run
   on EXP yet for end users, but it's what we build/test against now. Launch arg `-NoFirstLoginPopup`
   skips the EXP main-menu popup.

**Known dev-env gotchas (all observed live in #sml-development this week):**
- **Editor won't open from the Start Menu / "reopen last project."** Open by double-clicking
  `FactoryGame.uproject` instead. (SML is still investigating; a fix is reportedly building.)
- **`Bridge` engine plugin removed** (engine size cut) → `LogPluginManager: Error: Unable to load
  plugin 'Bridge'`. Disable it in the relevant `*.Target.cs` `DisabledPlugins` if it blocks the build.
- **`UnrealBuildTool.xml` copy / file-lock errors** (MSB3027/MSB3021, "user-mapped section open") →
  a reboot clears the lock.
- **Asset dumping with `-nullrhi` no longer works** (UE no longer loads mesh data under nullrhi). Use
  **`-RenderOffscreen`** instead.

## Part B — Convert SmartFoundations to a Game Feature Plugin

Per the guide, the Asset Manager now requires Game Features for auto-detected asset types
(**input contexts**, icon libraries, `FGUserSettings`) — Smart! uses these, so conversion is required,
not optional. ("Basically all mods that add content should switch; only pure libraries don't.")

1. Close the editor.
2. **Move the plugin** from `<Project>/Mods/SmartFoundations` to **`<Project>/Mods/GameFeatures/SmartFoundations`**.
   (Authoritative path per Alpakit `PackagePlugin.cs`: *"Game features have to stay under
   Mods/GameFeatures to be picked up by the game as such."* The guide text saying "GameFeatures/Mods"
   is a typo — verified against the repo, which has `Mods/GameFeatures/ExampleMod`.)
3. **Create a `GameFeatureData` asset** in the mod's content root, named **exactly** `GameFeatureData`.
4. Configure the GameFeatureData: **load/unload actions**, **primary-asset search paths**, and the
   content lists (icon libraries, input mapping contexts, etc.). Mirror the updated `ExampleMod`
   GameFeatureData (Arch ported it; commit `23f71dc`). All asset types that used to be auto-discovered
   by class path (icon libraries, **input contexts**, `FGUserSettings`) are now **primary assets** and
   must be discoverable through the Asset Manager — being a GFP with correct search paths is what makes
   that work.

**C++ GameFeature — RESOLVED (was the top unknown).** C++ mods convert to GFPs the same way: SirDigby's
`DigbyTool` (a C++ mod) is doing exactly this in #sml-development. Confirmed by the SML devs:
- **Plugin name does not change**; **no core redirects needed** (Archengius).
- **Alpakit auto-handles the pak path remap** (`Mods/<Mod>` -> `Mods/GameFeatures/<Mod>`); Mircea fixed
  the folder handling in Alpakit. The remap is Unreal staging + Alpakit adding the dir mapping.
- Alpakit has a **GFP template** ("make a new plugin with the alpakit template" to experiment), but
  **publishing GFPs is disabled for the initial 1.2 tools drop** because SMM/SMR can't distribute them
  yet — that's the gate on SML 3.12's public release. We can develop/test locally regardless.
- Bonus: **GFPs can register schematics, research trees, and most vanilla content WITHOUT SML** via the
  Asset Manager (Archengius). Potential future SML-dependency reduction — note, don't act on it during
  the port.
- GFPs are only hot-(un)loadable if pak-only and use no SML features; Smart! uses SML features, so it
  loads at startup like today (fine).

## Part C — Code port (the Smart!-specific work)

1. **Recompile against new headers** (game CL ~480321+, UE 5.6.1). Use the Phase-0 API-surface map
   (110 FG headers / 100 game types) as the per-break checklist. The `v3.11.3...dev` compare shows
   exactly which headers/signatures moved.
2. **`TObjectPtr<>` migration (major).** FactoryGame moved off raw `UObject*` to `TObjectPtr<>`. Raw
   pointers compile for now but risk GC crashes and become a hard error next SML. Force the check
   early via UHT config and fix incrementally:
   ```ini
   [UnrealHeaderTool]
   NonEngineNativePointerMemberBehavior=Disallow
   ```
   **Precise rule (confirmed by Archengius):** `TObjectPtr` is required **only on `UPROPERTY`
   declarations**. **Local variables and function arguments must stay raw `UObject*`** — do NOT convert
   those. So the migration is bounded to our `UPROPERTY` pointer *members*. Mostly drop-in (`TObjectPtr`
   auto-casts to raw), but **pointer-by-reference** params need manual changes. UE5 ships an optional
   conversion tool (unverified for mods — try + report back). Scope it from our `UPROPERTY(...) U*\*`
   members (a grep gives the exact count).
3. **Enhanced Input migration (hits our input system directly).**
   - Fields moved between `InputMappingContext` and `InputAction`. Recover old values from the **1.1
     backup project** (or FModel/Content Inspector on a shipped Smart! build) — they're lost on upgrade.
   - **Parent Context:** `mParentContext` moved `UFGInputMappingContext` → `UFGChildInputMappingContext`.
     Right-click the asset → "Convert to a Different DataAsset Type" → `FGChildInputMappingContext`.
     Child mapping contexts are **primary assets**, so they're auto-discovered **only when the mod is a
     GFP** (with correct search paths); otherwise register them in the Asset Manager manually.
   - **Known runtime bug (workaround in-hand):** child input mapping contexts can throw `LogAssetManager:
     Error: Registered PrimaryAssetId ..._ChildInputMappingContext ... does not match object's real id`,
     which breaks load. Fix: implement `UFGChildInputMappingContext::GetPrimaryAssetId()` (the
     #sml-development thread on 2026-05-30 has the exact override snippet — "avoid returning a valid
     primary asset id for the class CDO itself"). Re-check whether SML ships this fix before hand-rolling it.
   - One contributor noted Enhanced Input Actions effectively **need a C++ module** to wire up now —
     Smart! already has one, so we're ahead there.
   - **Player-mappable settings** (`Is Player Mappable`, Name, Display Name, Display Category) moved
     from the InputContext "Mappings" records onto **Input Actions**. The formerly-experimental
     "Player Mappable Key Settings" is now required.
4. **Recompile + Alpakit package** on 5.6.1 (game closed), then in-game smoke on EXP.

## Part D — Things to verify / watch (not blocking, but flagged)

- **SML 3.12 public release / SMM GameFeature distribution** — gates publishing; watch #toolkit-updates.
- **Content Tags** possibly now 1:1 in vanilla (Arch) — may let us drop SML's; confirm later.
- **Console shared codebase** — extra code/assets present; no action expected.
- **Custom Icon Libraries on servers** — base-game bug fixed in 1.2; no change expected on our side.
- **User Config Resetting** (SML 3.12 feature) — review which Smart! config options should be
  excluded from user reset (`Allow User Reset`); optional, no code change required to inherit it.
- **ficsit.app disclosures** — controller-compatibility + Network Activity/AI disclosure fields are
  now structured fields; uploads are blocked until answered. (We already have an AI disclosure.)

## Suggested order of attack
A (env) → B step-2/3 spike to settle the C++ GameFeature unknown → C1 (compile-fix vs headers) →
C2 (TObjectPtr) interleaved with C1 → C3 (Enhanced Input) → package + EXP smoke. Don't publish until
#toolkit-updates green-lights and SMM supports GameFeature distribution.
